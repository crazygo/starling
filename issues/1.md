## Current State

The star chart label rendering system (implemented in PR #21) uses a three-priority greedy placement algorithm to render star and constellation names on the canvas:

- **Group 1** (forced): Solar system objects whitelist
- **Group 2** (forced): Constellation/asterism names and their member stars' proper names
- **Group 3** (competitive): All other named stars in the viewport, sorted by magnitude, capped at 20

The placement algorithm tries 4 candidate positions per label (right-upper → left-upper → right-lower → left-lower) and uses bounding-rect collision detection to avoid overlap.

Currently there are **no automated tests** for this system. Correctness is verified only by manual visual inspection of the Vercel preview or local builds.

---

## Problem

Without automated tests, the following regressions are undetectable until someone notices visually:

1. A refactor breaks the greedy algorithm and Group 3 labels start overlapping Group 2 labels
2. A culture-switch bug causes Chinese asterism names to appear in western mode
3. The `HIP \d+` filter is accidentally removed and raw HIP identifiers appear on the chart
4. The 20-label cap for Group 3 is broken and performance degrades at high star density
5. Label positions shift unexpectedly after a projection change

---

## Candidate Solutions

### Option A — Unit tests for the placement engine (pure Dart, no Flutter)

Extract the label placement logic into a standalone class (e.g. `LabelPlacementEngine`) with no dependency on `Canvas` or `CustomPainter`. Write `dart test` unit tests against it.

**What to test:**
- Two overlapping candidates → only the brighter one is kept
- Group 2 labels always appear in the output even in an extremely crowded viewport
- Group 3 cap: given 100 candidate stars, output contains ≤ 20 Group 3 labels
- `HIP \d+` stars are never included in the output
- Fallback position selection: if right-upper is blocked, left-upper is chosen next

**Pros:** Fast, no rendering required, runs in CI with `flutter test`
**Cons:** Does not catch visual regressions (e.g. font size wrong, color wrong)

---

### Option B — Flutter Golden Tests

Use `matchesGoldenFile` to render the `StarChart` widget with a fixed mock dataset and compare pixel output against a committed baseline image.

**What to test:**
- Western culture: Orion viewport with expected label positions
- Chinese culture: northern sky with asterism names visible
- Zoom level change: labels update correctly

**Pros:** Catches visual regressions automatically
**Cons:** Golden files must be regenerated manually after intentional visual changes; can produce false positives due to font rendering differences across platforms (requires `--update-goldens` discipline)

---

### Option C — Integration with existing PR screenshot workflow (PR #18)

The existing `pr-screenshot.yml` Action already builds and screenshots the app on every PR. Extend it to navigate to a specific RA/Dec viewport known to contain Orion (western) or Beidou (Chinese) and upload as a named artifact.

**Pros:** Zero additional test code; reuses existing infrastructure; human-readable output
**Cons:** Still requires human review; no automated pass/fail signal for label correctness

---

## Recommendation

**Short term:** Implement Option A (unit tests for placement engine). This is the highest ROI — it catches logic regressions automatically with no visual rendering required, and can be added alongside PR #21.

**Medium term:** Add Option B (golden tests) once the label visual style is stable and no longer changing frequently.

**Long term:** Extend Option C with fixed-viewport screenshots for a human-readable regression baseline on every PR.
